## 1. 5-Level Planning Hierarchy

VeloCoach plans training in five levels, from strategic to tactical. Each level has a clear data owner, trigger, and output.

```
┌──────────────────────────────────────────────────────────────┐
│ Level 1: TRAINING PLAN (Makrozyklus)                         │
│ Goal + Event + Strategy → Phase structure + Block boundaries │
│ Created once per training cycle. Snapshot, read-only.        │
│ Tables: plans, blocks                                        │
├──────────────────────────────────────────────────────────────┤
│ Level 2: BLOCK PLAN (Mesozyklus)                             │
│ User refines upcoming block: constraints, schedule, focus    │
│ Updated before each block starts. Recurring touchpoint.      │
│ Table: blocks (user_inputs_json)                             │
├──────────────────────────────────────────────────────────────┤
│ Level 3: WEEKLY PLAN (Mikrozyklus)                           │
│ Concrete session slots: types, durations, TSS targets        │
│ Generated by LLM within block context. Cached.               │
│ Table: week_skeleton_cache                                   │
├──────────────────────────────────────────────────────────────┤
│ Level 4: SESSION PLAN (Einzeltraining)                       │
│ Detailed workout: intervals, cadence, ICU codes              │
│ Generated by LLM on explicit user trigger (cost control).    │
│ Table: planned_workouts                                      │
├──────────────────────────────────────────────────────────────┤
│ Level 5: CHAT ADJUSTMENTS                                    │
│ Ad-hoc changes to any level via conversational interface     │
│ Table: plan_adjustments                                      │
└──────────────────────────────────────────────────────────────┘
```

### 1.1 What is a Block?

A block is a mesocycle — the fundamental trainable unit within a phase. Typically 3–4 weeks: 2–3 load weeks followed by 1 deload week. Blocks are derived deterministically from the phase’s deload pattern:

- `every_3_weeks` → blocks of 3 weeks (2 load + 1 deload)
- `every_4_weeks` → blocks of 4 weeks (3 load + 1 deload)
- `none` (peak/taper) → entire phase is one block, no deload

Example: A 9-week Base phase with `every_3_weeks` produces 3 blocks:

```
Block 1: weeks 1–3 (load, load, deload)
Block 2: weeks 4–6 (load, load, deload)
Block 3: weeks 7–9 (load, load, deload)
```

The last block may be shorter if phase weeks don’t divide evenly.

### 1.2 Block as Recurring User Touchpoint

Before a block becomes active, the user is prompted:

- “Your next block starts Monday. Any changes?”
- Available days override, vacation weeks, constraints, focus notes
- These inputs are stored in `blocks.user_inputs_json` and override plan-level defaults for that block’s weeks.

In Phase 1 of implementation, blocks are created automatically and the user sees them in the timeline. The block-planning touchpoint (user edits before activation) is built in Phase 2.

### 1.3 LLM Cost Control

LLM calls are made only at Levels 3 and 4:

- **Level 3 (Weekly Skeleton):** Called when a week needs planning. Cached by input fingerprint. One LLM call per week.
- **Level 4 (Session Detail):** Called only when user explicitly triggers “build workouts” or “push to Intervals.icu.” Never automatic.

Level 1 and Level 2 planning is deterministic (pure TypeScript functions). Level 5 is LLM-powered but user-initiated.

## 2. System Architecture

```
┌─────────────────────────────────────────────────────────┐
│  FRONTEND (React/Vite)                                  │
│  Thin presentation layer. No planning logic.            │
│  Exception: generatePlanStructure() used client-side    │
│  for instant wizard preview (pure function, no DB).     │
│  All persistence via edge function calls.               │
└──────────────────────────┬──────────────────────────────┘
                           │ single calls
┌──────────────────────────▼──────────────────────────────┐
│  EDGE FUNCTIONS (Supabase/Deno)                         │
│  All orchestration, validation, LLM calls, sync.        │
│  Constitution enforcement lives here.                   │
│  create-plan is the primary orchestrator.               │
└──────────────────────────┬──────────────────────────────┘
                           │ reads/writes
┌──────────────────────────▼──────────────────────────────┐
│  DATABASE (Supabase Postgres)                           │
│  Source-of-truth tables + cache tables.                  │
│  RLS: every table filtered by auth.uid() = user_id.    │
└─────────────────────────────────────────────────────────┘
```

## 3. Planning Pipeline

### 3.1 Plan Creation (Level 1)

```
User fills PlanCreationWizard on /plan
  │
  ├─→ Client calls generatePlanStructure() for instant PREVIEW
  │   (pure function, no DB, no LLM — user sees phases + blocks immediately)
  │
  └─→ User confirms → single call to create-plan edge function
        │
        ├─→ Archive existing active plan
        ├─→ Call compute-athlete-context (refresh athlete_state)
        ├─→ Read athlete_state
        ├─→ Call generatePlanStructure() server-side
        ├─→ Derive blocks from phases + deload patterns
        ├─→ INSERT plans row (snapshot, read-only)
        ├─→ INSERT blocks rows (one per block)
        ├─→ Call generate-week-skeleton (current week only)
        └─→ Return { planId, planStructure, blocks, weekContext, weekSkeleton }
```

### 3.2 Block Planning (Level 2) — Phase 2 Implementation

```
Block approaching (status: upcoming)
  → User prompted: "Block 2 starts Monday. Review?"
  → User edits: available_days, constraints, vacation weeks, focus
  → UPDATE blocks.user_inputs_json
  → Invalidate week_skeleton_cache for block's weeks
  → Block status → active
```

### 3.3 Weekly Planning (Level 3)

```
generate-week-skeleton called (by create-plan or directly)
  → Reads: athlete_state, plan (from plans table via planId),
           block context (from blocks table)
  → Block user_inputs override plan-level defaults
  → Calls Anthropic API (Claude Sonnet 4.6)
  → Validates with validateWeekSkeleton()
  → Caches in week_skeleton_cache
  → Returns WeekSkeleton
```

### 3.4 Session Planning (Level 4) — User-Triggered

```
User triggers "Build Workouts" for a specific week
  → build-workouts edge function
  → Reads WeekSkeleton + AthleteState
  → Calls Anthropic API (Claude Sonnet 4.5)
  → Returns WorkoutSession[] → persisted to planned_workouts
```

### 4.2 Cache Tables

|Table                |Contains                  |Keyed By                 |Invalidated By                    |
|---------------------|--------------------------|-------------------------|----------------------------------|
|`athlete_state`      |Compiled AthleteState JSON|user_id (unique)         |Re-running compute-athlete-context|
|`week_skeleton_cache`|Validated WeekSkeleton    |user_id + week_start_date|Input fingerprint mismatch        |

### 4.3 Data Flow Rules

```
Current training setup     → athlete_preferences  (NOT plans)
Current fitness load       → wellness_days         (NOT plans.current_ctl)
Active plan structure      → plans WHERE status='active' → plan_structure_json
Block context for a week   → blocks WHERE plan_id=X AND start_date <= week AND end_date >= week
Connection credentials     → athlete_connections
```

### 4.4 The Snapshot Rule

`plans` rows are **write-once, read-many**:

- No UPDATE on `plans` rows after creation
- Changes go to `plan_adjustments`
- New plan = archive old (status=‘archived’, archived_at=now()) + INSERT new
- Snapshots capture ALL inputs + computed structure + constitution_version

### 4.5 Block Lifecycle

```
Plan created → all blocks inserted with status='upcoming'
First block → status='active' (matches current date)
Block ends → status='completed', next block → 'active'
User edits upcoming block → user_inputs_json updated (Level 2)
```

## 5. Edge Function Contracts

### create-plan (Orchestrator)

```
POST /create-plan
Input: {
  eventDemandProfile: string,
  eventName?: string,
  eventDate?: string,
  hoursPerWeek: number,
  availableDays: string[],
  strengthSessionsPerWeek?: number,
  entryState: "fresh_start" | "mid_training" | "returning_after_break",
  typology?: "PYRAMIDAL" | "POLARIZED" | "SS_THRESHOLD",  // absent = system default
  vacationWeeks?: string[]     // ISO dates of known vacation week Mondays
}
Output: {
  planId: string,
  planStructure: PlanStructure,
  blocks: Block[],
  weekContext: WeekContext,
  weekSkeleton: WeekSkeleton
}
Side effects:
  - Archives existing active plan
  - Refreshes athlete_state
  - Inserts plan + block rows
  - Caches first week skeleton
```

### generate-week-skeleton

```
POST /generate-week-skeleton
Input: {
  weekStartDate?: string,      // defaults to current Monday
  planId?: string              // reads plan + block from DB
}
Output: {
  weekSkeleton: WeekSkeleton,
  weekContext: WeekContext,
  blockContext: { blockNumber, phase, isDeloadWeek, weekInBlock },
  planSummary: { ... }
}
```

### compute-athlete-context

```
POST /compute-athlete-context
Input (optional): { specialWeekType?, loadCompleteness?, estimatedUntrackedLoad? }
Output: { state_json: AthleteState }
```

### build-workouts (User-triggered only)

```
POST /build-workouts
Input: { weekSkeleton, weekStartDate }
Output: { workouts: WorkoutSession[] }
```

## 6. Constitution Enforcement

Three levels of enforcement:

1. **`constitution.json`** — numeric constants injected into system prompts and TypeScript logic.
1. **`planningCore.ts`** — pure functions enforce phase durations (S8.1), load targets (S6.4), deload patterns, stress budgets (S4b).
1. **`validateWeekSkeleton()`** — post-LLM validation checks guardrails G1–G11.

The LLM generates; TypeScript validates. Never ship unvalidated LLM output to the database.

### The Plan Page Flow

```
/plan loaded
  ├─ No active plan → PlanCreationWizard (full page)
  │    Step 1: Goal & Event (profile, event, entry state)
  │    Step 2: Strategy & Schedule (hours, days, typology selection)

  │    Step 4: Confirm → create-plan edge function
  │
  └─ Active plan exists → Plan View
       ├─ Macrocycle Timeline (phases from plan_structure_json)

       ├─ Weekly Calendar (from week_skeleton_cache or fresh generation)
       └─ Side Panel (week summary / session detail)
```

### Typology Selection in Wizard Step 2

The system computes a recommended typology from `event_demand_profile × hours_per_week` using Constitution S4.1 + S4a.1. This recommendation is shown to the user as a default. The user can accept or override it.

Available typologies (Constitution S4):

- **Pyramidal** — balanced approach, system default for 6–10h/week
- **Polarized (80/20)** — for >10h/week with strong aerobic base
- **Sweet Spot / Threshold** — for ≤8h/week, FTP-focused goals

LINEAR is never user-selectable — it is system-imposed during re-entry only.

If the user’s selection conflicts with their hours (e.g., Polarized at 6h/week), a `TYPOLOGY_MISMATCH` soft warning is shown (Constitution S11.2). The user’s choice is respected and stored on the plan. See `docs/constitution-v7-amendments.md` for full mismatch conditions.

### Wizard Step 3: Client-Side Preview

The PlanCreationWizard imports `generatePlanStructure()` from `src/lib/coaching/planningCore.ts` (the re-export shim). This is a **pure function** — no DB access, no network calls. It computes the full phase and block structure instantly, so the user sees:

- Phase timeline with dates and durations
- Block boundaries within each phase
- Deload week positions
- Known vacation weeks highlighted

The user can adjust phase durations (±1–2 weeks within Constitution S8.1 guardrails). On confirm, the server recomputes and persists (never trusting client computation for storage).

## 8. Block Derivation Algorithm

Blocks are derived deterministically in `create-plan` from the PlanStructure output. The algorithm:

```typescript
function deriveBlocks(plan: PlanStructure): BlockRecord[] {
  const blocks: BlockRecord[] = [];
  let blockNumber = 0;

  for (const phase of plan.phases) {
    const deloadPattern = phase.deloadStrategy.defaultPattern;

    if (deloadPattern === "none") {
      // Peak/Taper: entire phase is one block, no deload
      blockNumber++;
      blocks.push({
        phase: phase.phase,
        blockNumber,
        blockNumberInPhase: 1,
        startDate: phase.startDate,
        endDate: phase.endDate,
        weeks: phase.weeks,
        loadWeeks: phase.weeks,
        deloadWeekNumbers: [],
      });
      continue;
    }

    // Base/Build: split by deload interval
    const interval = deloadPattern === "every_3_weeks" ? 3 : 4;
    let weeksCovered = 0;
    let blockInPhase = 0;

    while (weeksCovered < phase.weeks) {
      blockNumber++;
      blockInPhase++;
      const remaining = phase.weeks - weeksCovered;
      const blockWeeks = Math.min(interval, remaining);
      const blockStart = addWeeks(phase.startDate, weeksCovered);
      const blockEnd = addWeeks(blockStart, blockWeeks) - 1 day;

      // Deload is last week of block (if block is full-length)
      const deloadWeeks = blockWeeks === interval
        ? [phase.weekNumbers[weeksCovered + blockWeeks - 1]]
        : []; // Partial block: no deload

      blocks.push({
        phase: phase.phase,
        blockNumber,
        blockNumberInPhase: blockInPhase,
        startDate: blockStart,
        endDate: blockEnd,
        weeks: blockWeeks,
        loadWeeks: blockWeeks - deloadWeeks.length,
        deloadWeekNumbers: deloadWeeks,
      });

      weeksCovered += blockWeeks;
    }
  }
  return blocks;
}
```

This is pseudocode — the actual implementation adapts to the existing `computeDeloadWeeks()` output in `planningCore.ts`.
